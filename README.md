# npmsafe

npmsafe — CLI-утиліта для аналізу npm-пакетів на ознаки supply-chain атак. Інструмент читає `package.json`, виконує AST-аналіз JavaScript-файлів та формує консолідований звіт про ризики. Разом із цим репозиторій містить інфраструктуру для роботи з AI-асистентом (OpenAI Assistants API), що допомагає планувати подальший розвиток.

## Можливості CLI
- **AST-евристики**: виявлення `eval`, `new Function`, `require('child_process')`, деструктуризації та викликів `child_process` (у тому числі через псевдоніми).
- **Перевірка lifecycle-скриптів**: підсвітка небезпечних `postinstall`, `preinstall`, `install`, `prepare` зі скачуванням файлів, shell-командами тощо.
- **Оцінка ризику**: рівні `safe`, `low`, `suspicious`, `malicious` з ваговою системою, що враховує сумарну суворість проблем.
- **Формати звіту**: текстовий режим за замовчуванням або JSON для інтеграції в CI/CD.
- **Суворий режим**: прапорець `--strict` підвищує будь-який `low`/`suspicious` до помилки (`exit code 2`).

## Встановлення
```bash
npm install -g npmsafe
```
або локально в репозиторій з запуском через `npx`:
```bash
npm install npmsafe --save-dev
npx npmsafe --help
```

Рекомендована версія Node.js — **>=18** (у CI використовується Node 20).

## Використання
```bash
# Аналіз каталогу (package.json шукається автоматично)
npmsafe .

# Перевірка конкретного package.json
npmsafe ./packages/example/package.json

# JSON-звіт
npmsafe . --format json

# Детальна інформація та суворий режим
npmsafe . --verbose --strict
```

### Коди завершення
| Рівень ризику | Exit code | Пояснення |
| ------------- | --------- | --------- |
| `safe`        | 0         | Небезпечних патернів не знайдено |
| `low`         | 1 (2 зі `--strict`) | Низький ризик: lifecycle без небезпечних команд, дрібні попередження |
| `suspicious`  | 1 (2 зі `--strict`) | Виявлено середню загрозу, потрібен ручний аудит |
| `malicious`   | 2         | Критичні ознаки компрометації |
| `error`       | 3         | Внутрішня помилка виконання |

## Що перевіряється
- **Lifecycle-скрипти**: мережеві завантаження (`curl`, `wget`), shell-команди, видалення файлів, зміна прав доступу.
- **`child_process`**: імпорт через `require`/`import`, присвоєння у змінні, деструктуризація (`const { exec } = require('child_process')`), виклики через псевдоніми (`cp.exec()`, `run()` після деструктуризації).
- **Динамічне виконання коду**: `eval`, `new Function`, великі/важкопарсингові файли.
- **Статистика**: кількість перевірених файлів, знайдених скриптів і AST-попереджень.

## Приклад текстового звіту
```
🔍 Аналіз безпеки пакету: suspicious-package
📊 Рівень ризику: 🟠 SUSPICIOUS

⚠️  Виявлені проблеми:
1. script: Підозрілий postinstall скрипт: Завантаження файлів з мережі
   Розташування: package.json:scripts.postinstall
2. ast: Виконання системних команд через exec()
   Розташування: malicious.js:3

📈 Статистика:
   Перевірено файлів: 3
   Підозрілих скриптів: 1
   AST-попереджень: 1
```

## Розробка
1. Клонувати репозиторій і встановити залежності:
   ```bash
   git clone <repo-url>
   cd npmsafe
   npm install
   ```
2. Запустити автотести:
   ```bash
   npm test
   ```
   Тести створюють тимчасові пакети, проганяють аналізатор і перевіряють критичні сценарії (`eval`, lifecycle-скрипти, псевдоніми `child_process`).
3. Додавайте нові евристики разом із регресійними тестами у `test/test.js`.

## AI-асистент (OpenAI Assistants API)
Репозиторій містить сценарії для роботи з AI-розробником, який читає документацію в `docs/` і пропонує плани розвитку.

### Підготовка
1. Створіть `.env` на основі `.env.example` і задайте `OPENAI_API_KEY`.
2. Запустіть `npm run bootstrap` — скрипт створить асистента та завантажить документи в File Search.
3. Запустіть `npm run plan` — асистент прочитає PRD/архітектуру/модель загроз і згенерує план робіт.

> У GitHub Actions додайте секрет `OPENAI_API_KEY`, щоб автоматизувати планування.

### Чому саме Assistants API
- Codex депрекейтед; GPT-4o/Reasoning-моделі підтримуються через Assistants API.
- Асистент зберігає інструкції, читає файли та може використовувати інструменти.

### Налаштування моделей
У `.env` можна перевизначити:
- `OPENAI_ASSISTANT_MODEL` — модель для покоління коду (наприклад, `gpt-4o`).
- `OPENAI_REASONING_MODEL` — для архітектурних роздумів (наприклад, `o4-mini`).

## Структура проєкту
```
npmsafe/
├── docs/                    # PRD, архітектура, модель загроз, roadmap
├── prompts/                 # Промпти для AI
├── scripts/                 # Скрипти взаємодії з Assistants API
├── src/                     # Реалізація CLI-аналізатора
├── test/                    # Інтеграційні тести аналізатора
└── .github/                 # Шаблони issues/PR та CI
```

## CI
- `.github/workflows/ci.yml` запускає `npm ci` і `npm test` для `push`/`pull_request`.
- Додатковий job (`assistant-plan`) виконується лише якщо задано секрет `OPENAI_API_KEY` і генерує план від AI.
